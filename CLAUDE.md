# langchaint

A provider-neutral LLM client library. Alpha: the API is unstable and may change without notice.

## Docstrings and comments

Docstrings and comments are self-contained for human readers: state the constraint or reasoning inline, never cite internal dev documents, decision logs, spec files, or dead alternatives to the live code.
They also never refer to the repo's own prior state. Diff-relative prose is a sentence that only makes sense to a reader who saw the change that introduced it: wording like "as before", "no longer", or "now", a reference to a state that exists nowhere in the current tree, or a justification for a question no reader of the file would ask (often marked by one element of a group carrying a defense its siblings lack). The test: a sound sentence reads the same whether the code was born in its current shape or arrived there by refactor.
A grep-able phrase list enforced by `scripts/CI.sh` was considered and rejected: the motivating instances carried no temporal phrase while sanctioned sentences about current behavior do ("SDK events the package does not model are dropped"), so a phrase grep misfires in both directions; enforcement is review judgment.

Module docstrings are the spec of record for mechanics. This file states each design tenet and the deliberate absences code cannot show; how an implemented behavior works (wire encodings, marker budgets, cancellation semantics, edge cases) lives in the docstring of the code that implements it, and a change to such a behavior updates that docstring, not this file. Rejected alternatives and their one-line reasons are deliberate absences too and stay in this file beside the rule they shield: a rejected alternative has no code to carry a docstring, and its recorded reason is the re-litigation shield for the reviewer's design objections (rule 8 in `.claude/agents/commit-reviewer.md`), so condensing this file must never sweep one into deletion (a docs-only provider fact inside such a reason moves to the docstring of the code that would depend on it).

Document what a function raises. Every function whose body can raise (directly, by re-raising a caught error, or by propagating a documented raise from a helper it calls) names the exception types a caller may see and the condition for each, in a `Raises:` section when the raise is direct or in prose when it is not. This is a review convention, not a lint: the DOC501/DOC502 ruff rules are de-selected because they misread `raise type(exc)(...)` and `raise Class.factory(...)` (see the note in `pyproject.toml`), so they cannot catch a missing or stale raise.

## Provider facts

Never assert provider behavior (wire parameters, usage-field semantics, exception taxonomies, cache rules) from memory; verify it against the installed `anthropic`/`openai` packages by introspection before writing or reviewing code that depends on it.

A verified fact's home is the docstring of the code that depends on it: the adapter module docstrings' "Verified against" sections, or the docstring of the specific function the fact shapes, with the SDK version when the fact could drift.
CLAUDE.md records only facts that justify a rule spanning modules, cited inside the rule that depends on them or in the verified-provider-facts bullet of the Design rules.
A fact only one module needs stays at the code, not in this file.

## Naming rules

- All names explicit: `system_prompt` not `system`, `inference_params` not `params`, `tool_manager` not `tools`.
- The keyword and the variable passed to it share one name: a call site reading `name_1=name_2` gives one concept two names, so a `ToolManager` is passed as `tool_manager=tool_manager`. `ToolManager` keeps its name despite the low-information head "Manager": every candidate drops one of its two jobs (`ToolRegistry`/`ToolSet` drop dispatch, `ToolDispatcher` drops schemas).
- One name per concept end to end; no aliases. The class is `Provider` even where prose says "adapter"; the fix for that tension, if any, is prose discipline, not a rename.
- Neutral over provider vocabulary: when providers disagree, take the majority wire name or a neutral one, never a single provider's term (`ToolCall` not `ToolUse`, `Part` not `ContentBlock`, `StopReason` with `"end_turn"`).
- Bare `input_tokens` is banned: anthropic's field of that name excludes cache reads while openai's equivalent includes them. Use the partition `input_tokens_cache_read`/`_cache_write`/`_cache_none` and the derived `input_tokens_total`.
- Units and encodings in names: `cost_in_usd`, `elapsed_seconds`, `backoff_base_seconds`, `*_usd_per_million_tokens`; the `_json` suffix (`args_json`) marks unparsed JSON text.
- Family prefixes keep related fields adjacent in sorts and completions: `input_tokens_*`, `backoff_*`, `generate_one`/`generate_many` (arity in the suffix).
- No stutter with the holder (`tool.name`, not `tool.tool_name`); cross-object references carry the full name (`tool_call_id` on `ToolMessage`).
- The plain noun goes to the interface: `Tool` is the protocol `ToolManager` holds and an application implements to add its own tool form, because the protocol name is read in signatures, type errors, and docs far more often than any concrete name is written. The concrete forms are named by the technology their argument spec is written in (`PydanticTool` with `args_model`, `JSONSchemaTool` with `args_schema`): the class name carries how the arguments are specified, the field name carries what it holds. `CaptureTool` is named for its fixed behavior instead: its argument spec is pydantic like `PydanticTool`'s, so the technology does not distinguish it; the absent function does. Rejected: a capability name for the protocol (`DispatchableTool`: every tool dispatches, so the qualifier excludes nothing); field names mirroring the class (`pydantic_model`/`raw_schema`: they leave the `args_*` family, orphaning `args_json`, and turn the `ToolSchema` pass-through into `args_schema=self.raw_schema`, one concept under two names); `ArgsSchemaTool` for the JSON-schema form (it claims as distinguishing the args-schema concept both forms emit on `ToolSchema`); `RawSchemaTool` (says how the schema is treated, where `JSONSchemaTool` says what the constructor takes).
- `cache_breakpoint` is the neutral name for a user-placed prompt-cache boundary, where the providers' wire names disagree (anthropic block-level `cache_control`, openai part-level `prompt_cache_breakpoint`): True on a `TextPart`/`ImagePart` means the reusable prompt prefix ends at that part.
- `max_attempts` counts requests sent including the first, so 1 means no retrying. Adapters store a `with_options(max_retries=0)` copy of the SDK client so the SDK never retries beneath the package and the count stays true.
- `content`, `output`, and `reasoning` are three concepts, never one word. `content` is a model-facing message body, `MessageContent = str | Sequence[Part]`, the field both SDKs name `content` on the wire, carried by `UserMessage`/`ToolMessage`/`ToolOutputExplicit`. `output` is the generation result payload `OutputT` on `Response.output`/`ProviderResult.output`, the assistant text or the SDK-parsed `response_format` instance; the two coincide only for plain text, since a structured `output` is a parsed `BaseModel`, never a `Part`. `reasoning` is opaque provider round-trip data carried as `ReasoningTrace` elements in `AssistantMessage.turn`, never a `Part` and never in a user or tool body. At the tool layer, `ToolOutput` names what a tool function returns (`MessageContent | ToolOutputExplicit[AppDataT]`), split from the generation-layer `output` the way `Response` and `ProviderResult` are split by layer. The storage forms, coercions, and payload encodings are in the `messages.py` and `tools.py` docstrings.

## Design rules

- Generation only via binding: `LLM` has no generate methods; `LLM.bind(...) -> BoundLLM[OutputT]` freezes everything that determines the cacheable prompt prefix, and changing parameters is `rebind(...)`, never a per-call override. `automatic_prompt_caching` is a required keyword of `bind` with no default: caching changes billing, so the library never chooses a caching configuration for the user, and any convenience on top of user-stated caching (such as `generate_many`'s `warm_cache`) is opt-in and default-off. A `False` default was proposed and rejected: an unstated `False` silently opts every high-volume binding out of caching, a billing choice as real as opting in, so every call site states the boolean.
- Prompt caching is the bound `automatic_prompt_caching` plus user-placed `cache_breakpoint` marks, honored under either binding value; bound False with no marks caches nothing. The wire mechanics (the anthropic 4-marker budget, `cache_ttl`, and marker placement rules; the openai latest-N server rule; the system-prompt parts rendering) are in the two adapter docstrings, and `warm_cache`'s semantics are on `BoundLLM.generate_many`. A per-part TTL is rejected until someone asks: it adds an ordering-validation surface for a pattern the uniform `cache_ttl`'s two-binding workaround already covers (the mixing-TTLs ordering rule and its docs source are in the anthropic adapter's `cache_ttl` docstring).
- The application owns the tool loop; langchaint ships no library-owned agent loop, and a tool function returns data, never a control-flow signal: stop, route, escalate, and needs-approval are decisions the app makes in its own loop between turns, reading `app_data` or `is_error`, so a control-flow return channel (a goto or an engine-state update smuggled through a tool's return value) is forbidden. The tool forms (`PydanticTool`, `JSONSchemaTool`, `CaptureTool`), the `Tool` protocol, and the `dispatch`/`dispatch_many` semantics are in `tools.py`. `CaptureTool`, the function-free form whose `capture` returns the validated instance as the required `DispatchCaptured.captured`, is how an application receives a structured final response in a `tool_choice="required"` loop; the capture is data, and whether it ends the loop stays the application's decision. `JSONSchemaTool.dispatch` validates the arguments against `args_schema` with `jsonschema` after the JSON-object precondition, mapping failures onto neutral `InvalidToolArgsDetail` values so every tool form fails into the same `DispatchInvalidToolArgs` arm. An app-supplied `args_validator` parameter is rejected: it saved the app nothing (the app still imported `InvalidToolArgsDetail` and wrote the jsonschema mapping itself), and a semantic rule the schema cannot express belongs in the tool function, returning `ToolOutputExplicit` with `is_error=True`. Rejected on `InvalidToolArgsDetail`: a machine-readable `code` field (pydantic's `type` and jsonschema's `validator` are library-specific vocabularies, so the field is a passthrough posing as neutral, and no consumer branches on failure kind) and a `source: object` passthrough of the raw library item (readable only with provenance knowledge of which tool form produced it, and an app needing the raw jsonschema errors reruns the validation itself). There is no concurrency-limit parameter.
- One `RateLimiter` (`rate_limiter.py`) owns retrying and pacing; the name states what it guards, the provider's rate limit, not the retrying it also owns. It is stateful and shareable, one instance being one shared budget for the account it guards, and its slot gates every request-start path. There is deliberately no `requests_per_minute`: an in-flight bound self-adjusts throughput along request duration. Rate-limit errors (what `Provider.classify` returns `"rate_limit"` for) gate admission account-wide through the pause/probe cycle described in `rate_limiter.py`; other transient errors pause nobody.
- Adapters wrap official SDK clients and delegate stream assembly and structured-output parsing to the SDK. No hand-written wire TypedDicts.
- The error taxonomy splits two orthogonal properties: the retry axis (the loop in `llm.py` retries a `TransientError` and propagates everything else, so "non-retriable" is a concept, not a class) and the batch axis (`AbortBatchError` cancels the sibling requests, while a `GenerationError` becomes one item's failure row). A structured parse that returns no parsed output is classified refusal, truncation, or transient, never silently-wrong data, and the two structured-only leaves (`RefusalError`, `MaxCompletionTokensExceededError`) are per-item terminal results, not batch-abort. The classes, their billing carriers, and the not-retried-by-default policies are in `exceptions.py` and the adapters' `_parsed_output`.
- `Usage` carries exactly two kinds of data, and this is the criterion for any future field: provider-invariant counters (a new provider either fits the three-way input partition or does not belong in `Usage`), and `cost_in_usd`, the adapter-priced scalar that absorbs provider-variant billing structure. Provider-specific detail in counts stays on `usage_raw`, the raw SDK usage riding beside every `Usage`; `usage` is never `None` on a carrier (a non-billing attempt is `ZERO_USAGE`, and `usage_raw is None` is the "no wire payload" signal).
- The cost breakdown is a reporting projection one layer up from `Usage`, computed on demand and never stored on a carrier: a per-category cost is derived (counts times rates) and does not fold like a counter, so it fails the `Usage` field criterion. `pricing.py` holds the neutral SDK-free arithmetic, and each backend's public `cost_breakdown(usage_raw, pricing)` routes through the same `price` call as the stored `cost_in_usd`, so a reported breakdown and the stored scalar cannot drift. `cache_ttl="1h"` paired with a table lacking `cache_write_1h_usd_per_million_tokens` is checked at adapter construction; a statically-required rate (a required field on the shared `PricingTable`, or an anthropic-specific table subclass) was considered and rejected: it forces a fabricated rate into every table that can never see a 1-hour write, and the constraint is value-dependent, binding only when `cache_ttl` is `"1h"`, so the construction-time guard checks the pair where both values are in hand.
- Usage scope: `usage` is the paid total across every attempt of the call, on both `Response` and `GenerationError`, so the default a caller bills on is the money actually spent; `Response.usage_successful_attempt` is the single kept answer's own usage. Both fold from `attempt_records`, the one source of truth, so they cannot desync; `to_row` columns and the tracing mapper's span usage are paid totals likewise.
- Bedrock support comes from the SDKs' bundled clients, not from a Converse adapter: `anthropic_bedrock_model` takes the Bedrock wire model id (`AnthropicBedrockModelName`) and sends it verbatim, with per-id Bedrock API and pricing routing in `ANTHROPIC_BEDROCK` (documented in `langchaint/anthropic/__init__.py`), and OpenAI on Bedrock is `AsyncBedrockOpenAI` passed as `client`.
- OpenAI support is the Responses API only (`OpenAIResponsesProvider`); there is no Chat Completions adapter. Third-party chat-completions-compatible servers (vLLM, Ollama) are out of scope.
- Streams yield `StreamItem = str | ToolCall`, nothing else: no tool-call delta items (partial argument JSON is not actionable) and no usage or stop items (`usage` and `stop_reason` live on `final()`'s `Response`).
- No rewrap, no copy: never take data out of an SDK object only to reconstruct it in a package object of the same shape. Pass provider values through by reference; construct a package model only where the shape genuinely changes (normalizing `Usage`, deriving `StopReason`).
- Discriminate outcomes by type, not by a nullable flag: a function with several outcomes returns a union of frozen dataclasses, one arm per outcome, and the arm that carries extra data types it as a required non-optional field, so matching the arm narrows the payload and no consumer writes `cast`, `assert x is not None`, or `# type: ignore` to read a field the discriminator already guarantees is present. Split into arms only where the fields genuinely differ: two outcomes that carry the same fields and differ by a boolean stay one arm holding that boolean. The rule governs a value a caller receives and branches on, not an element type gathered into a homogeneous collection and consumed by folding; it also governs the payload type, keeping a concrete type parameter where the caller knows it (the worked example, `PydanticTool`'s `AppDataT` against `ToolManager`'s erased channel, is in `tools.py`).
- Reasoning is preserved verbatim across turns: each reasoning element becomes one `ReasoningTrace` in `AssistantMessage.turn`, and the consuming adapter re-emits every trace in place, unconditionally, with no `provider_name` tag and no adapter-side drop. Trimming is the app's job, by rebuilding concluded assistant turns without their traces; there is no bind-time on/off parameter because it would be redundant with editing the conversation. The motivations, payload encoding, and per-provider replay conditions are in the `ReasoningTrace` docstring and the two adapter docstrings.
- Honor user inputs faithfully, even when they are invalid, and make no promise about how a provider will respond: provider-side rules are the provider's to enforce. Never probe or test an endpoint to learn what error an invalid input combination raises, and never add client-side guards guessing at provider-side rules: send the request and surface the provider's error. Do not restate this per case: a rule or comment saying a particular invalid input draws a particular provider error and that the adapter surfaces it is nothing but an instance of this rule and does not belong (a verified provider fact cited as the reason a design avoids sending some input is a different thing and stays). Client-side raises are reserved for documented provider facts (the 4-marker limit, the tool_result marker placement) and for defects that would otherwise produce a silently wrong result.
- Pydantic only where serde plus validation pay for themselves: the message tree (what a caller persists to store a conversation), `Usage` (whose `NonNegativeInt` counters guard the openai subtraction-derived partition), and tool args models (which validate model-produced argument JSON). Everything else is a frozen dataclass or NamedTuple; each qualifying model's docstring states what its validation buys.
- The SDKs are optional dependencies the application pins directly; the hard dependencies are `pydantic` and `jsonschema`. langchaint declares no extras of its own: a langchaint extra could name only a base SDK distribution, so it would misdirect any application that needs an SDK variant the base does not pull (`anthropic[bedrock]`, `openai[bedrock]`), and pinning the SDK is the application's to do. jsonschema stays a hard dependency: `JSONSchemaTool` lives in the neutral core, where no import path separates its users from `PydanticTool` users, so making it optional would surface as a `ModuleNotFoundError` at dispatch time instead of at install. The neutral core imports no SDK, and each backend subpackage imports its SDK at module top, guarded so a missing package raises a `ModuleNotFoundError` naming the package to install. The import path is the boundary: only code that reaches for a backend requires its SDK, each backend subpackage holds everything for that provider (catalog and concrete adapter), and the neutral base contract stays in `langchaint.provider` (singular).
- Each backend subpackage holds at least one function returning a ready `LLM` (`anthropic_model` and `anthropic_bedrock_model` in `langchaint.anthropic`, `openai_model` in `langchaint.openai`), its `Literal` model-name type, and its pricing table, with `client`, `pricing`, and `rate_limiter` defaulting sensibly and overridable. Each constructor is named for its selector, its own catalog's `model_name`, not the `LLM` it returns; `anthropic_bedrock_model`'s catalog is the Bedrock wire model ids (`AnthropicBedrockModelName`), sent verbatim so the id in application code, on the wire, and in traces is one string, and the first-party names are not accepted as aliases. The signatures are deliberately asymmetric where the providers differ; each parameter's meaning and each asymmetry's reason is documented on the function. Models outside the catalog are built directly from the re-exported concrete adapter: `LLM(AnthropicMessagesProvider(...))`. Prices are the one provider fact not verifiable by SDK introspection; each subpackage docstring carries the source URL.
- The public surface is tiered by audience: applications import the neutral core from top-level `langchaint` and each backend's constructor, names, pricing, concrete adapter, and `cost_breakdown` from its subpackage; adapter authors import the provider contract from `langchaint.provider`. Top-level `__all__` re-exports only the SDK-free application surface (re-exporting a backend would force `import langchaint` through both SDKs); internal helpers and the adapter-author contract stay importable from their own modules but off it.
- `InferenceParams` (`inference_params.py`) is deliberately minimal: only `max_completion_tokens`, `reasoning_effort`, and `temperature`, with no `top_p` or `seed`; None leaves the provider default in place. An unmapped provider parameter is reached by subclassing the adapter, never a passthrough dict.
- OTel tracing ships in-tree as `langchaint.tracing` (opentelemetry-api only, pinned directly by the application), thin, following the guarded-import subpackage pattern and off the top-level `__all__`. Governing premises: the wrapper never fakes an event boundary a span is supposed to measure; the mapper owns only attribute names and values, and never receives the conversation; a telemetry failure is caught and logged, never propagated, whether it comes from a raising mapper or from serializing captured content; wrapping is unconditional, and enabling, disabling, or routing tracing is OTel SDK configuration, never application code.
The mapper is a function a caller passes, defaulting to the public `gen_ai_attributes`. Selecting it by a format name instead is rejected: the name set has one member and buys only a spelling for a function that is already public, and a second vocabulary ships as another public function, which composes (`{**gen_ai_attributes(r), **other_attributes(r)}`) where a name set does not.
Recording the conversation on spans is `capture_message_content`, a required keyword with no default on every Traced class, applied by the wrapper (which already holds the conversation as a method argument) and never by the mapper, so the mapper premise above still holds. Content on by default is rejected: it reverses the leak-safety position, ships conversations from every already-traced application on upgrade, and contradicts the convention's own SHOULD NOT. Emitting the frozen prefix once per binding with a digest on each span, instead of repeating the system prompt and tool definitions on every span, is rejected for now: it is the right answer when span size shows up on the tracing bill, since that prefix is byte-identical across every span a binding opens, but it needs a backend-side join and fails the thin-wrapper tenet.
The convention's payload JSON schemas are vendored under `tests/semconv_genai` at a recorded commit, because nothing publishes them, and every emitted payload is validated against them; the mechanics, the per-attribute exemptions, and the limits of what those schemas enforce are in `tests/test_tracing.py`. Tightening that validation by selecting each part's `$defs` arm from its own `type` instead of letting the schema's `anyOf` fall through to `GenericPart` is rejected: the emitted part shapes are already pinned by exact-equality assertions that catch a renamed type or a dropped field, and `_content_parts` renders an `ImagePart` with its bytes dropped, a part that matches only the `GenericPart` arm because `BlobPart` requires both a `modality` and the content that rendering deliberately omits, so arm selection would fail that rendering to re-check what another assertion already covers.
A `langchaint.*` span attribute is reserved for what the GenAI convention defines no counterpart for, which is the criterion the prefix means; where a convention key exists it wins, including for values whose measurement origin differs slightly (`gen_ai.response.time_to_first_chunk` is measured from the span's start, so it also covers the `RateLimiter` slot wait, and the docstring states that rather than keeping a private key over it). Tool dispatch spans come from `TracedToolManager`, a `ToolManager` subclass rather than a TracedLLM-style wrapper: `LLM.bind` types `tool_manager` nominally, so only a subclass serves `bind` and the traced loop as one object, and `dispatch_many` inherits per-call spans through `self.dispatch` without restating its settle-and-group semantics. The span model, retry-visibility events, and pinned GenAI convention revision are in `tracing/__init__.py`.
- Verified provider facts that cross module boundaries (checked against anthropic 0.116.0 / openai 2.45.0): both SDK clients retry internally by default (`DEFAULT_MAX_RETRIES = 2`) and honor `retry-after-ms`/`retry-after` only up to 60 seconds; both SDKs' `APIStatusError` exposes `.response` (an httpx.Response) whose headers carry retry-after; `with_options(max_retries=0)` returns a client copy on all five client classes (`AsyncAnthropic`, `AsyncOpenAI`, `AsyncBedrockOpenAI`, `AsyncAnthropicBedrock`, and `AsyncAnthropicBedrockMantle`). Single-module wire facts (usage-field shapes, cache-write splits and per-request caching parameters, reasoning block shapes and replay conditions, `store`/`include` behavior) live at the code per the Provider facts section, in the adapter module docstrings and the docstrings of the functions they shape.

## Module map

One line per module; the list changes rarely, so it does not rot the way a signature snapshot would.

- `llm.py`: `LLM` composes a provider adapter and a `RateLimiter`; `LLM.bind(...)` freezes the cacheable prompt prefix into a `BoundLLM`.
- `provider.py`: dual-audience. It holds the adapter-author contract (`Provider`/`BoundProvider`/`ProviderStream`/`ProviderResult`/`Binding`/`ErrorClass`) alongside four types that stay on the application surface (`StreamItem`/`ToolChoice`/`SpecificToolChoice`/`PricingTable`). It is the neutral base contract and imports no SDK; the concrete adapters live in the backend subpackages.
- `rate_limiter.py`: `RateLimiter`, the single owner of retrying and pacing.
- `exceptions.py`: the whole error vocabulary: `TransientError`, `AbortBatchError`, `AttemptRecord`, the `GenerationError` base and its leaves (`RetriesExhaustedError`, `RefusalError`, `MaxCompletionTokensExceededError`), `InvalidToolArgsError`, `DispatchExceptionGroup`, and `StreamProtocolError`.
- `response.py`: the generate result `Response`, plus `to_row` for flattening a `Response` or a `GenerationError` to one row shape.
- `streaming.py`: `StreamHandle`, an async iterator of stream items whose `final()` returns the `Response`.
- `tools.py`: `PydanticTool` (pydantic `args_model`), `JSONSchemaTool` (raw JSON schema, arguments validated against it with jsonschema), and `CaptureTool` (function-free capture of a validated `args_model` instance) unified by the `Tool` protocol, `ToolManager` (per-call `dispatch` and the concurrent `dispatch_many`), the tool return union `ToolOutput` and its explicit arm `ToolOutputExplicit`, the dispatch outcome types (`DispatchHandled`/`DispatchInvalidToolArgs`/`DispatchUnknownTool`, union `DispatchOutcome`; `DispatchCaptured`, returned only by `CaptureTool.capture`), and `InvalidToolArgsDetail`, the neutral per-failure detail `DispatchInvalidToolArgs` carries as `details`.
- `messages.py`: the provider-neutral message tree and content parts; `AssistantMessage` stores the ordered `turn` of `ReasoningTrace`/`TextPart`/`ToolCall` elements.
- `usage.py`: `Usage`, the token-count partition plus the adapter-priced `cost_in_usd`; `ZERO_USAGE`, its additive identity.
- `checked_copy.py`: `CheckedCopyModel`, the base of the package's pydantic models, whose `model_copy` raises on update keys that are not model fields.
- `pricing.py`: `PriceableCounts`, `CostBreakdown`, and `price`, the neutral cost arithmetic every stored `cost_in_usd` and every backend `cost_breakdown` routes through; imports no SDK and no error class.
- `anthropic/`: the anthropic backend subpackage, everything that needs the anthropic SDK in one place. `__init__.py` holds `anthropic_model`, `anthropic_bedrock_model`, `ANTHROPIC_PRICING`, `ANTHROPIC_BEDROCK`, `BedrockRouting`, `AnthropicModelName`, `AnthropicBedrockModelName` and re-exports the adapter and `CacheTTL`; `messages_provider.py` holds the concrete adapter `AnthropicMessagesProvider`. Importing the subpackage requires the anthropic SDK.
- `openai/`: the openai backend subpackage, everything that needs the openai SDK in one place. `__init__.py` holds `openai_model`, `OPENAI_PRICING`, `OpenAIModelName` and re-exports the adapter; `responses_provider.py` holds the concrete adapter `OpenAIResponsesProvider`. Importing the subpackage requires the openai SDK.
- `inference_params.py`: `InferenceParams` and `ReasoningEffort`.
- `tracing/`: the OTel tracing subpackage, everything that needs opentelemetry-api in one place. `__init__.py` holds `TracedLLM`/`TracedBoundLLM`/`TracedStreamHandle`/`TracedToolManager`, the `AttributeMapper`/`SpanAttributes` types, and `gen_ai_attributes`, the default mapper, public so a custom mapper extends it instead of restating its keys. Importing the subpackage requires opentelemetry-api; it is off the top-level `__all__` so `import langchaint` stays OTel-free.

## Checks

Run `scripts/CI.sh`; it is the CI check that must pass, and it runs `pyrefly check`, `ruff check`, and `pytest` through `uv run` (so the tools resolve from the locked dev dependency group, not a hand-activated `.venv`), all with zero errors. DOC201 is deliberately de-selected (it forces restating return types in prose), and DOC501/DOC502 are de-selected because they misresolve the raise shapes this package uses (see "Document what a function raises" above and the `pyproject.toml` note); the other docstring rules are kept and satisfied by writing the docstrings. The tests are offline: they feed constructed SDK objects into the adapter helpers and drive `BoundLLM`/`StreamHandle` with stub providers, so they need no API keys. `tests/*` carries a `SLF001` per-file-ignore because the tests exercise private helpers and attributes (`Provider._request`, `BoundLLM._bound_provider`) directly.

# Commit Review

After each commit lands, spawn the `commit-reviewer` agent (`.claude/agents/commit-reviewer.md`) with the commit sha as its prompt. The agent file holds the reviewer's rules 1-8 (scope, no re-running checks, spec fidelity, depth, test quality, user experience, output, design objections); this section holds the author's side of the loop. The reviewer model, passed as the model override at spawn, scales with the commit's stakes: Opus for serious work (new features, behavior changes, anything open to design objections under rule 8 in the agent file), Sonnet only for the most trivial docstring changes.

Done gate: a commit counts as Done only after its review has returned, any confirmed issues are fixed, and every raised objection is resolved. One feature stays one commit: during the review -> correct loop, fixes amend into the reviewed commit (`git commit --amend`, force-pushing if it was already pushed), and the amended commit is re-reviewed. Design objections block the gate: the author resolves each one when the review returns, adopting it by editing the rule or code its premise challenges, or rejecting it by recording in `CLAUDE.md` beside the rule it shields the one-line reason for keeping the design as written (a rejected alternative has no code to carry a docstring), which becomes the re-litigation shield rule 8 honors. Deferring an objection to a later commit is not resolving it. That edit rides in the same amend; there is no separate open-items file to keep in sync. A not-Done commit is never handed off as the finished result or built on by the next unit of plan work; the loop's own amend-and-force-push of a not-yet-Done commit is how it reaches Done, not a violation.

# Code smells

Whenever you find yourself writing the following code smells, ask "how can I redesign and refactor to avoid them?"

- `**kwarg` unpacking
- `Any` for typing
- `cast` as a general solution. Treat `cast` like Rust's `unsafe`: not banned, but never a reflex reach. The default is a redesign that makes the type flow without it. A `cast` is warranted only at a genuine boundary where a deliberately-opaque value re-enters a typed API and its shape is known-true by construction (it came from that same API's own serialization), the alternative is worse (for example a revalidation that silently reshapes the payload), and it is contained to that one line with a comment stating why the shape holds. Outside that narrow case, a `cast` is the smell to redesign away.
