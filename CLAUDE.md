# langchaint

A provider-neutral LLM client library. Prototype: nothing is stable, everything can be rewritten.

## Docstrings and comments

Docstrings and comments are self-contained for human readers: state the constraint or reasoning inline, never cite internal dev documents, decision logs, spec files, or dead alternatives to the live code.

Document what a function raises. Every function whose body can raise (directly, by re-raising a caught error, or by propagating a documented raise from a helper it calls) names the exception types a caller may see and the condition for each, in a `Raises:` section when the raise is direct or in prose when it is not. This is a review convention, not a lint: the DOC501/DOC502 ruff rules are de-selected because they misread `raise type(exc)(...)` and `raise Class.factory(...)` (see the note in `pyproject.toml`), so they cannot catch a missing or stale raise.

## Provider facts

Never assert provider behavior (wire parameters, usage-field semantics, exception taxonomies, cache rules) from memory; verify it against the installed `anthropic`/`openai` packages by introspection before writing or reviewing code that depends on it.

## Naming rules

- All names explicit: `system_prompt` not `system`, `inference_params` not `params`, `tool_manager` not `tools`.
- Call sites read `exact_name=exact_name`: the keyword is named after what the value is, so a `ToolManager` is passed as `tool_manager=tool_manager`.
- One name per concept end to end; no aliases.
- Neutral over provider vocabulary: when providers disagree, take the majority wire name or a neutral one, never a single provider's term (`ToolCall` not `ToolUse`, `Part` not `ContentBlock`, `StopReason` with `"end_turn"`).
- Bare `input_tokens` is banned: anthropic's field of that name excludes cache reads while openai's equivalent includes them. Use the partition `input_tokens_cache_read`/`_cache_write`/`_cache_none` and the derived `input_tokens_total`.
- Units and encodings in names: `cost_in_usd`, `elapsed_seconds`, `backoff_base_seconds`, `*_usd_per_million_tokens`; the `_json` suffix (`args_json`) marks unparsed JSON text.
- Family prefixes keep related fields adjacent in sorts and completions: `input_tokens_*`, `backoff_*`, `generate_one`/`generate_many` (arity in the suffix).
- No stutter with the holder (`tool.name`, not `tool.tool_name`); cross-object references carry the full name (`tool_call_id` on `ToolMessage`).
- `max_attempts` counts requests sent including the first, so 1 means no retrying. Adapters store a `with_options(max_retries=0)` copy of the SDK client so the SDK never retries beneath the package and the count stays true.
- `content`, `output`, and `reasoning` are three concepts, never one word. `content` is a model-facing message body, `MessageContent` = `str | Sequence[Part]` (aliased in `messages.py`), carried by `UserMessage`/`ToolMessage`/`ToolOutputExplicit` and every adapter wire content helper, matching the field both SDKs name `content` on the wire; the pydantic message fields spell out the frozen-tuple form (`str | tuple[Part, ...]`) because they coerce the sequence to a frozen tuple rather than aliasing it. `AssistantMessage` carries no `content` field: it stores the ordered `turn: tuple[TurnElement, ...]` (`TurnElement = ReasoningTrace | TextPart | ToolCall`, emission order) and derives the `.text`/`.tool_calls` views; the assistant turn's re-feedable text is its `TextPart` elements, read as `.text`. `UserMessage` and `AssistantMessage` take their single argument positionally (`UserMessage("hi")`, `AssistantMessage("hey")`), and a bare string `turn` coerces to one `TextPart` on every construction path. `output` is the generation result payload `OutputT` on `Response.output`/`ProviderResult.output` (and the adapter's `_parsed_output`), the assistant text or the SDK-parsed `response_format` instance. `content` and `output` coincide only for plain text: a structured `output` is a parsed `BaseModel`, not a `Part`, so it can never appear in a message body, while the same turn's re-feedable text lives separately on `Response.assistant_message`. `reasoning` is opaque provider round-trip data, distinct from both: it is never a `Part` and never in a user or tool body (OpenAI Responses models it as a separate output item and the OTel GenAI convention as a token counter, neither as content); it is carried as `ReasoningTrace` elements in `AssistantMessage.turn`, each `reasoning` the producing SDK item's `model_dump(mode="python", exclude_none=True)` dict, and it is a dict, not a JSON string (unlike `args_json`), because neither provider's reasoning wire form is a string. At the tool layer, `ToolOutput` names what a tool function returns, the union `MessageContent | ToolOutputExplicit[AppDataT]`; it is distinct from the generation-layer `output`/`OutputT` the same way `Response` and `ProviderResult` are split by layer.

## Design rules

- Generation only via binding: `LLM` has no generate methods; `LLM.bind(...) -> BoundLLM[OutputT]` freezes everything that determines the cacheable prompt prefix. No per-call parameter overrides; changing parameters is `rebind(...)`.
- The application owns the tool loop; langchaint ships no library-owned agent loop. A tool function returns data (`str | Sequence[Part]`, or a `ToolOutputExplicit` wrapping that plus `is_error` and `app_data`) and nothing that steers control flow: stop, route, escalate, and needs-approval are decisions the app makes in its own loop between turns, reading `app_data` or `is_error`. Tool content is model-facing, so it is exactly `str | Sequence[Part]`, the same type as `ToolMessage.content`: it must already be in a form the model reads, so a function with a typed result serializes it to that form itself (a single `TextPart`/`ImagePart` is returned as a one-element `Sequence[Part]`, since a bare `Part` is not a `Sequence[Part]`). Content carries no `BaseModel`/`Mapping` arm and `dispatch` does no serialization: model-facing content is not the place for a typed object, unlike the `app_data` channel, which is app-facing (the model never sees it) and does carry a typed `BaseModel` or `Mapping` through untouched. A control-flow return channel is forbidden (as in LangGraph's `Command`, whose `goto` reroutes the loop and `update` mutates engine-owned state through a leaf tool's return value): it is the symptom of moving the loop out of the app, which langchaint does not do.
- One `RateLimiter` (`rate_limiter.py`) owns retrying and pacing: `max_attempts`, `backoff_base_seconds`, `backoff_max_seconds`, `max_in_flight`. It is stateful and shareable: one instance passed to several `LLM`s is one shared budget for the account they hit. Its slot gates every request-start path and backoff sleeps outside the slot. There is deliberately no `requests_per_minute`: an in-flight bound self-adjusts throughput along request duration.
- Rate-limit errors gate admission account-wide. `Provider.classify` returns `"rate_limit"` for them (anthropic 429 and 529, openai 429), which lands on `TransientError.is_rate_limit`; a server-stated retry-after also counts as rate-limit evidence. The limiter then pauses admission for everyone sharing it: for the server-stated wait when one was sent (capped at 60 seconds so an erroneous or hostile header cannot stall the client indefinitely), else for the failing task's backoff delay. After the pause, admission stays limited to one probe request at a time until a completed request records success. Other transient errors (timeouts, 5xx) pause nobody; the failing task's own backoff covers them.
- Adapters wrap official SDK clients and delegate stream assembly and structured-output parsing to the SDK. No hand-written wire TypedDicts.
- A structured parse that returns no parsed output is classified three ways by the adapter's `_parsed_output`. A provider-signaled refusal (anthropic `stop_reason == "refusal"`, openai a `ResponseOutputRefusal` content block) raises `RefusalError`; a truncation at the token cap (anthropic `stop_reason == "max_tokens"`, openai `status == "incomplete"` with `incomplete_details.reason == "max_output_tokens"`) raises `ExceededMaxCompletionTokensError`; every other empty parse raises `TransientError`, whose exhaustion surfaces `RetriesExhaustedError`, never silently-wrong data. `RefusalError` and `ExceededMaxCompletionTokensError` are per-item terminal results, not batch-abort: each becomes a `to_row` failure row like `RetriesExhaustedError` and leaves siblings running. Neither is retried by default: an app whose economics differ on refusals overrides `_parsed_output` (the "subclass the adapter" escape hatch), and the token-cap fix is a larger `max_completion_tokens` via `rebind`. The text path never raises for these: a capped or refused text turn is a legitimate `Response` the app inspects by `stop_reason` (`"max_tokens"` / `"refusal"`); the two errors fire only on the structured path.
- The error taxonomy splits two orthogonal properties. The retry axis (read by the loop in `llm.py`): the loop retries a `TransientError` and propagates everything else, so "non-retriable" is a concept, not a class. The batch axis (read by `generate_many`): `AbortBatchError` (what `Provider.classify` returns `"abort"` for) cancels the sibling requests, while a `GenerationError` becomes one item's failure row. `AbortBatchError` and every `GenerationError` leaf are non-retriable and differ only on this batch axis. `GenerationError` is the base for the three terminal per-item results (`RetriesExhaustedError`, `RefusalError`, `ExceededMaxCompletionTokensError`); it carries `attempt_records`, `model`, `provider_name`, `elapsed_seconds`, and `stop_reason`, and derives `usage`/`cost_in_usd` from the records. The adapter that detects a refusal or truncation raises the leaf carrying only the one rejected 200's billing; the retry loop records that attempt and re-raises the enriched leaf, so no unenriched leaf reaches a caller. `AttemptRecord` carries `usage`/`cost_in_usd` (the attempt's billing, `None` for a transport failure) so a rejected 200's cost survives on the failure row. All exceptions live in the bottom-level `exceptions.py`, which imports only `StopReason` and `Usage` (both leaf modules) and stays acyclic; `response.py` imports the error and record types from it.
- Bedrock support comes from the SDKs' bundled clients (`AsyncAnthropicBedrock`, `AsyncBedrockOpenAI`), not from a Converse adapter.
- OpenAI support is the Responses API only (`OpenAIResponsesProvider`); there is no Chat Completions adapter. Third-party chat-completions-compatible servers (vLLM, Ollama) are out of scope until someone asks.
- Streams yield `StreamItem = str | ToolCall`, nothing else. Each `ToolCall` is yielded once, complete, when its block closes; there are no tool-call delta items because partial argument JSON is not actionable. There are no usage or stop items: usage, `cost_in_usd`, and `stop_reason` live on `final()`'s `Response`.
- No rewrap, no copy: never take data out of an SDK object only to reconstruct it in a package object of the same shape. Pass provider values through by reference; construct a package model only where the shape genuinely changes (normalizing `Usage`, deriving `StopReason`).
- Discriminate outcomes by type, not by a nullable flag: a function with several outcomes returns a union of frozen dataclasses, one arm per outcome, and the arm that carries extra data types it as a required non-optional field, so matching the arm narrows the payload and no consumer writes `cast`, `assert x is not None`, or `# type: ignore` to read a field the discriminator already guarantees is present. Split into arms only where the fields genuinely differ: two outcomes that carry the same fields and differ by a boolean stay one arm holding that boolean, not two field-identical arms. The rule governs a value a caller receives and branches on, not an element type gathered into a homogeneous collection and consumed by folding (filter, count, positional read). The rule also governs the payload type: `Tool` carries an `AppDataT` type parameter (defaulting to `None`) that the checker solves from the function's `ToolOutputExplicit` return, and `Tool.dispatch` returns `DispatchHandled[AppDataT]`, so an app that dispatched a known single tool reads `app_data` at its concrete type with no `isinstance`. The type is `AppDataT | None` because `app_data` is genuinely optional; a plain `is not None` guard is not the banned type-narrowing. `ToolManager.dispatch` holds a heterogeneous tool set whose per-call `AppDataT` is erased, so it surfaces `DispatchHandled[BaseModel | Mapping[str, object] | None]` and the `isinstance` on `app_data` there is the intended heterogeneous fold.
- Reasoning is preserved verbatim across turns. Each reasoning element the model produces (an anthropic `thinking`/`redacted_thinking` block, an openai `reasoning` item) becomes one `ReasoningTrace` in `AssistantMessage.turn`, in emission order; the consuming adapter re-emits every trace in place, unconditionally. A trace is replayable only through the provider that produced it: re-fed through the other provider, its dict's type key is outside that API's input union, and the resulting provider error is the intended loud failure of a wrong-provider replay (deliberate provider switching first rebuilds concluded assistant turns without their traces, the same mechanism as trimming). There is no `provider_name` tag on the trace and no adapter-side drop of another provider's traces: that drop was a silent trim, which the control-surface rule below reserves for the app, and it contradicted the surface-the-provider-error policy this rule ends with. Three motivations: quality (a reasoning model that cannot see its prior reasoning across the app-owned tool loop re-derives or contradicts itself), correctness (Anthropic 400s a tool-use continuation whose latest turn's thinking blocks are missing or modified; OpenAI's stateless replay under `store=False` requires output items re-sent in their original order, with the reasoning payload in `encrypted_content`, which the API populates only under `include=["reasoning.encrypted_content"]`, sent on every request), and prefix caching (reasoning sits inside the growing cached prefix, so hitting the cache across a tool loop requires it present and byte-identical every turn). The payload is one opaque dict per element, the SDK item's `model_dump(mode="python", exclude_none=True)`, re-fed on consume by a single contained `cast` to the SDK param and appended unchanged, never neutralized field-by-field (the two providers' reasoning shapes are disjoint and the string fields must round-trip byte-identical) and not a JSON string (neither provider's reasoning wire form is a string, unlike `args_json`). Full reasoning history is the default and the conversation is the only control surface: trimming is the app's job, by rebuilding concluded assistant turns without their traces; a turn whose tool calls still await results must keep its reasoning, and there is no bind-time on/off knob because it would be redundant with editing the conversation. Replay is coupled to the binding: Anthropic rejects thinking re-fed on a request whose binding enables no reasoning, and the adapter surfaces that provider error rather than silencing it.
- Pydantic only where serde plus validation pay for themselves. The qualifying models are: the message tree (`UserMessage`/`AssistantMessage`/`ToolMessage`/`TextPart`/`ImagePart`/`ToolCall`/`ReasoningTrace`), which validates the content and turn unions and coerces lists to tuples at the user-facing construction boundary and is what a caller persists to store a conversation (`AssistantMessage` is the ordered `turn` union; `TurnElement`'s members have disjoint field sets, so re-validation selects the member by field match without a discriminator); `Usage`, whose `NonNegativeInt` counters and partition cross-check validator guard the openai subtraction-derived partition; and tool args models, which validate model-produced argument JSON. Everything else is a frozen dataclass or NamedTuple.
- The SDKs are optional dependencies, one extra per backend (`langchaint[anthropic]`, `langchaint[openai]`, `langchaint[all]`); only `pydantic` is a hard dependency. The neutral core imports no SDK, so `import langchaint` needs neither package. Each backend lives in its own subpackage (`langchaint.anthropic`, `langchaint.openai`) that imports its SDK at module top, guarded so a missing package raises a `ModuleNotFoundError` naming the extra to install. The import path is the boundary: only code that reaches for a backend requires its SDK, and a type checker following `langchaint.openai` never resolves anthropic's types (so the catalog is split by backend, not shared). Each backend subpackage holds everything for that provider, the model catalog and the concrete adapter (`langchaint.anthropic.messages_provider`, `langchaint.openai.responses_provider`); the neutral base contract stays in `langchaint.provider` (singular) and imports no SDK.
- Each backend subpackage holds one function returning a ready `LLM` (`anthropic_model` in `langchaint.anthropic`, `openai_model` in `langchaint.openai`), its `Literal` model-name type (`AnthropicModelName`/`OpenAIModelName`), and its pricing table (`ANTHROPIC_PRICING`/`OPENAI_PRICING`). Each function takes the provider's own model identifier as a `Literal` (the wire string is the model's only name), `client` defaulting to the native SDK client built from environment credentials (Bedrock routing is passing a Bedrock client), `pricing` defaulting to the model's public prices looked up in the backend's table and overridable by passing a `PricingTable` of your own rates, and `rate_limiter`. The signatures are not symmetric: `anthropic_model` takes a fifth keyword `default_max_completion_tokens: int = 4096` that `openai_model` lacks, because the Messages API requires `max_tokens` and its adapter fills a `None` `InferenceParams.max_completion_tokens` with this default, while the Responses API needs no such default. Models outside the catalog are built directly from a concrete adapter re-exported by the backend subpackage: `from langchaint.anthropic import AnthropicMessagesProvider; LLM(AnthropicMessagesProvider(...))`. Prices are the one provider fact not verifiable by SDK introspection; each subpackage docstring carries the source URL.
- The public surface is tiered by audience: applications import the neutral core from top-level `langchaint` and each backend's constructor, names, pricing, and concrete adapter from its subpackage (`langchaint.anthropic`/`langchaint.openai`); adapter authors import the provider contract from `langchaint.provider` (singular: `Provider`/`BoundProvider`/`ProviderStream`/`ProviderResult`/`Binding`/`ErrorClass`). Top-level `__all__` re-exports only the SDK-free application surface, so the provider constructors, pricing tables, and concrete adapters are off it (re-exporting them would force `import langchaint` through both SDKs). Internal helpers (`Admission`, `SequenceNotStr`, `ToolSchema`) and the adapter-author contract stay importable from their own modules but off it.
- `InferenceParams` (`inference_params.py`) is deliberately minimal: only `max_completion_tokens` and `reasoning_effort`, with no `temperature`, `top_p`, or `seed`. An unmapped provider knob is reached by subclassing the adapter, never a passthrough dict.
- `to_row(Response | GenerationError) -> dict[str, RowValue]` (`response.py`) flattens many results and errors into uniform rows for logging or dataframes; a success and any terminal failure (retries exhausted, refusal, truncation) fill the same keys.
- OTel tracing ships in-tree as `langchaint.tracing` under a `langchaint[otel]` extra (opentelemetry-api only; the SDK is a dev dependency for the tests' in-memory exporter), thin and content-free, following the guarded-import subpackage pattern and off the top-level `__all__`. The governing premise: the wrapper never fakes an event boundary a span is supposed to measure. So `TracedStreamHandle` iterates (recording `langchaint.time_to_first_token_seconds` and closing the span on a failing or abandoned stream) rather than pass through the iterator it needs to witness. `TracedBoundLLM.generate_many` is the deliberate exception: it delegates to `BoundLLM.generate_many` under one aggregate INTERNAL span carrying only `langchaint.batch_item_count`, and leaves per-item cost and timing to the returned rows (`to_row`); the mapper is not invoked for a batch. Span name (`{operation} {model}`, operation `"chat"`), kind, and status are wrapper-owned; only attribute names and values are mapper-owned, resolved once from `attribute_format: AttributeFormat | AttributeMapper` (default `"gen_ai"`). Kind is CLIENT for a span that wraps one outbound call (`generate_one`, the stream span) and INTERNAL for `generate_many`'s aggregate span, which makes no call of its own. The mapper receives `Response[object] | GenerationError` and never the conversation, so no built-in mapper can leak a prompt (content capture requires subclassing `TracedBoundLLM`); it is called only under `span.is_recording()`, and a raising mapper is caught and logged on the `langchaint.tracing` logger, never propagated, so a telemetry bug never discards a paid result. Wrapping is unconditional (`TracedLLM` mirrors `bind`/`rebind` so a rebound object stays traced); enabling, disabling, or routing tracing is OTel SDK configuration, never application code. Retry visibility is span events (`langchaint.attempt_failed` per failed record), not child spans. GenAI convention keys are verified against the pinned `opentelemetry-semantic-conventions` revision named in the module comment (0.64b0), never asserted from memory; the three-way cache partition stays whole under `langchaint.*` because the convention has no `cache_none` counterpart. `pyrefly` includes `tests` on its `search-path` so `tests.test_tracing` reuses `tests.test_bound_llm`'s fakes.
- Verified provider facts (checked against anthropic 0.116.0 / openai 2.45.0): anthropic `input_tokens` excludes cache tokens, openai Responses usage `input_tokens` includes `input_tokens_details.cached_tokens` and `.cache_write_tokens`; anthropic splits cache writes into 5m/1h (`CacheCreation`); openai bills cache writes starting with GPT-5.6; on gpt-5.6+ openai caching is configurable per request (`prompt_cache_options` with `mode: "implicit" | "explicit"` and `ttl: "30m"`; explicit mode with no breakpoints disables caching) and input text/image parts take `prompt_cache_breakpoint`, up to four written per request; the Responses API stores responses server-side unless `store=False` is sent; both SDK clients retry internally by default (`DEFAULT_MAX_RETRIES = 2`) and honor `retry-after-ms`/`retry-after` only up to 60 seconds; both SDKs' `APIStatusError` exposes `.response` (an httpx.Response) whose headers carry retry-after; `with_options(max_retries=0)` returns a client copy on all four client classes. Reasoning shapes (introspection-verified): anthropic `ThinkingBlock` is `{signature, thinking}` and `RedactedThinkingBlock` `{data}`, with `ThinkingBlockParam`/`RedactedThinkingBlockParam` members of the `MessageParam.content` union; openai `ResponseReasoningItem` is `{id, summary, content, encrypted_content, status}`, a `response.output` union member, and `ResponseReasoningItemParam` a `ResponseInputItemParam` member; `"reasoning.encrypted_content"` is a `ResponseIncludable` member and `include` is a parameter of `responses.create`/`.parse`/`.stream`. Reasoning runtime behavior: anthropic requires the latest assistant turn's thinking blocks re-sent unmodified in a tool-use continuation (else 400), filters prior turns' blocks itself, cannot have consecutive thinking blocks rearranged, and rejects thinking re-fed on a request whose binding enables no reasoning; openai requires output items between the last user message and the function-call outputs replayed in original order, and populates `encrypted_content` only when the request carries `include=["reasoning.encrypted_content"]` (`store=False` alone returns it None). OTel GenAI semconv 0.64b0 defines no reasoning message part, only the counter `gen_ai.usage.reasoning.output_tokens`.

## Module map

One line per module; the list changes rarely, so it does not rot the way a signature snapshot would.

- `llm.py`: `LLM` composes a provider adapter and a `RateLimiter`; `LLM.bind(...)` freezes the cacheable prompt prefix into a `BoundLLM`.
- `provider.py`: dual-audience. It holds the adapter-author contract (`Provider`/`BoundProvider`/`ProviderStream`/`ProviderResult`/`Binding`/`ErrorClass`) alongside four types that stay on the application surface (`StreamItem`/`ToolChoice`/`SpecificTool`/`PricingTable`). It is the neutral base contract and imports no SDK; the concrete adapters live in the backend subpackages.
- `rate_limiter.py`: `RateLimiter`, the one retry-budget and pacing knob.
- `exceptions.py`: the whole error vocabulary: `TransientError`, `AbortBatchError`, `AttemptRecord`, the `GenerationError` base and its leaves (`RetriesExhaustedError`, `RefusalError`, `ExceededMaxCompletionTokensError`), `InvalidToolArgsError`, and `StreamProtocolError`.
- `response.py`: the generate result `Response`, plus `to_row` for flattening a `Response` or a `GenerationError` to one row shape.
- `streaming.py`: `StreamHandle`, an async iterator of stream items whose `final()` returns the `Response`.
- `tools.py`: `Tool`/`ToolManager`, the tool return union `ToolOutput` and its explicit arm `ToolOutputExplicit`, and the dispatch outcome types (`DispatchHandled`/`DispatchInvalidToolArgs`/`DispatchUnknownTool`, union `DispatchOutcome`).
- `messages.py`: the provider-neutral message tree and content parts; `AssistantMessage` stores the ordered `turn` of `ReasoningTrace`/`TextPart`/`ToolCall` elements.
- `usage.py`: `Usage`, the token-count partition.
- `anthropic/`: the anthropic backend subpackage, everything that needs the anthropic SDK in one place. `__init__.py` holds `anthropic_model`, `ANTHROPIC_PRICING`, `AnthropicModelName` and re-exports the adapter; `messages_provider.py` holds the concrete adapter `AnthropicMessagesProvider`. Importing the subpackage requires the anthropic SDK.
- `openai/`: the openai backend subpackage, everything that needs the openai SDK in one place. `__init__.py` holds `openai_model`, `OPENAI_PRICING`, `OpenAIModelName` and re-exports the adapter; `responses_provider.py` holds the concrete adapter `OpenAIResponsesProvider`. Importing the subpackage requires the openai SDK.
- `inference_params.py`: `InferenceParams` and `ReasoningEffort`.
- `tracing/`: the OTel tracing subpackage, everything that needs opentelemetry-api in one place. `__init__.py` holds `TracedLLM`/`TracedBoundLLM`/`TracedStreamHandle`, the `AttributeFormat`/`AttributeMapper`/`SpanAttributes` types, and the built-in `"gen_ai"` mapper. Importing the subpackage requires opentelemetry-api (install `langchaint[otel]`); it is off the top-level `__all__` so `import langchaint` stays OTel-free.

## Checks

Run `scripts/CI.sh`; it is the CI check that must pass, and it runs `pyrefly check`, `ruff check`, and `pytest` through `uv run` (so the tools resolve from the locked dev dependency group, not a hand-activated `.venv`), all with zero errors. DOC201 is deliberately de-selected (it forces restating return types in prose), and DOC501/DOC502 are de-selected because they misresolve the raise shapes this package uses (see "Document what a function raises" above and the `pyproject.toml` note); the other docstring rules are kept and satisfied by writing the docstrings. The tests are offline: they feed constructed SDK objects into the adapter helpers and drive `BoundLLM`/`StreamHandle` with stub providers, so they need no API keys. `tests/*` carries a `SLF001` per-file-ignore because the tests exercise private helpers and attributes (`Provider._request`, `BoundLLM._bound_provider`) directly.

# Commit Review

After each commit lands, spawn one clean-context review agent for that commit. The agent has no memory of the writing session.
Use Sonnet as a reviewer for simple work, Opus for work that is more open to design objections (see point 8 below).

1. Scope: exactly one commit. Read ONLY from the git object store: `git show <sha>`, `git diff <sha>^ <sha>`, and `git show <sha>^:<path>` for surrounding context (for the root commit, `git show --root <sha>`). Never read the working tree or `HEAD`, which may have moved since the commit. Begin the review by printing the resolved hash, subject, and parent hash as proof of scope. Every finding must be about this commit's diff, not other commits, not the working tree, not the broader plan; design objections (rule 8) are exempt and may challenge a premise the commit inherits.
2. Do not re-run checks. `scripts/CI.sh` (`pyrefly check`, `ruff check`, and `pytest`) passed at zero errors on a green tree immediately before the commit was created, so re-execution is redundant, including the tests. Use static reading plus cheap read-only greps only.
3. Spec fidelity: before reading the diff, read the rules in `CLAUDE.md`, and state what the code should do. Then check the commit against your own reading, not the author's stated intent.
4. Depth: review correctness of code, adherence to the naming rules (`exact_name=exact_name`, explicit names), faithfulness of any stated provider facts, and consistency between the diff and the documents it edits (`CLAUDE.md`, `README.md`). Provider-behavior claims must be verified per the "Provider facts" section above, never asserted from memory.
5. Test quality: judge whether the added tests would catch a plausible wrong implementation, and whether they could be stronger. For adapter code, prefer tests that feed a constructed SDK object and inspect the normalized result over tests that mock the package's own types.
6. User experience: look for regressions in what application code has to write, especially added verbosity at public methods and constructors. Judge each public call site the commit changes by writing out the before and after forms; flag any that got longer or noisier without a stated reason.
7. Output: a report of confirmed issues, each with the evidence that confirms it, not a list of suspicions. Ground each finding in a concrete input and the wrong output it produces, or the exact `CLAUDE.md` sentence it violates. An empty report is a valid result.
8. Design objections (optional, non-blocking): after the confirmed issues, the reviewer may challenge the commit's design premises (SDK-delegation boundary, bind-only generation, the neutral ToolChoice vocabulary, retry ownership, usage normalization). Each objection must name (a) the specific premise it challenges, (b) the concrete cost if that premise is wrong, and (c) an alternative. Omit objections missing any of the three; do not restate preferences or re-litigate a premise the Design rules already record with their reasoning unless naming new evidence.
9. Done gate: a commit counts as Done only after its review has returned and any confirmed issues are fixed. The fix lands as a new commit, which is itself reviewed. Design objections do not block the gate; when the review returns, the author answers each objection inline where the premise lives, in the relevant Design rule in `CLAUDE.md`: adopting it edits that rule, rejecting it appends the one-line reason for keeping the rule as written, which becomes rule 8's re-litigation shield. That edit rides in the fix commit or a follow-up docs commit; there is no separate open-items file to keep in sync. Do not start the next unit of plan work on top of a commit that is not Done.

# Code smells

Whenever you find yourself writing the following code smells, ask "how can I redesign and refactor to avoid them?"

- `**kwarg` unpacking
- `Any` for typing
- `cast` as a general solution. Treat `cast` like Rust's `unsafe`: not banned, but never a reflex reach. The default is a redesign that makes the type flow without it. A `cast` is warranted only at a genuine boundary where a deliberately-opaque value re-enters a typed API and its shape is known-true by construction (it came from that same API's own serialization), the alternative is worse (for example a revalidation that silently reshapes the payload), and it is contained to that one line with a comment stating why the shape holds. Outside that narrow case, a `cast` is the smell to redesign away.
- Unpacking data from one container (list, dict, pydantic object) and back into another.
