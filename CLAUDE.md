# langchaint

A provider-neutral LLM client library. Alpha: the API is unstable and may change without notice.

## Docstrings and comments

Docstrings and comments are self-contained for human readers: state the constraint or reasoning inline, never cite internal dev documents, decision logs, spec files, or dead alternatives to the live code.

Document what a function raises. Every function whose body can raise (directly, by re-raising a caught error, or by propagating a documented raise from a helper it calls) names the exception types a caller may see and the condition for each, in a `Raises:` section when the raise is direct or in prose when it is not. This is a review convention, not a lint: the DOC501/DOC502 ruff rules are de-selected because they misread `raise type(exc)(...)` and `raise Class.factory(...)` (see the note in `pyproject.toml`), so they cannot catch a missing or stale raise.

## Provider facts

Never assert provider behavior (wire parameters, usage-field semantics, exception taxonomies, cache rules) from memory; verify it against the installed `anthropic`/`openai` packages by introspection before writing or reviewing code that depends on it.

## Naming rules

- All names explicit: `system_prompt` not `system`, `inference_params` not `params`, `tool_manager` not `tools`.
- Call sites read `exact_name=exact_name`: the keyword is named after what the value is, so a `ToolManager` is passed as `tool_manager=tool_manager`.
- One name per concept end to end; no aliases.
- Neutral over provider vocabulary: when providers disagree, take the majority wire name or a neutral one, never a single provider's term (`ToolCall` not `ToolUse`, `Part` not `ContentBlock`, `StopReason` with `"end_turn"`).
- Bare `input_tokens` is banned: anthropic's field of that name excludes cache reads while openai's equivalent includes them. Use the partition `input_tokens_cache_read`/`_cache_write`/`_cache_none` and the derived `input_tokens_total`. `Usage` also carries `output_tokens` and `output_tokens_reasoning` (the reasoning share), and the adapter-priced `cost_in_usd`; the raw SDK usage rides beside every `Usage` as `usage_raw: BaseModel | None` on each carrier for lossless recovery of provider-specific counts.
- Units and encodings in names: `cost_in_usd`, `elapsed_seconds`, `backoff_base_seconds`, `*_usd_per_million_tokens`; the `_json` suffix (`args_json`) marks unparsed JSON text.
- Family prefixes keep related fields adjacent in sorts and completions: `input_tokens_*`, `backoff_*`, `generate_one`/`generate_many` (arity in the suffix).
- No stutter with the holder (`tool.name`, not `tool.tool_name`); cross-object references carry the full name (`tool_call_id` on `ToolMessage`).
- `cache_breakpoint` is the neutral name for a user-placed prompt-cache boundary, where the providers' wire names disagree (anthropic block-level `cache_control`, openai part-level `prompt_cache_breakpoint`): True on a `TextPart`/`ImagePart` means the reusable prompt prefix ends at that part.
- `max_attempts` counts requests sent including the first, so 1 means no retrying. Adapters store a `with_options(max_retries=0)` copy of the SDK client so the SDK never retries beneath the package and the count stays true.
- `content`, `output`, and `reasoning` are three concepts, never one word. `content` is a model-facing message body, `MessageContent` = `str | Sequence[Part]` (aliased in `messages.py`), carried by `UserMessage`/`ToolMessage`/`ToolOutputExplicit` and every adapter wire content helper, matching the field both SDKs name `content` on the wire; the pydantic message fields spell out the frozen-tuple form (`str | tuple[Part, ...]`) because they coerce the sequence to a frozen tuple rather than aliasing it. `AssistantMessage` carries no `content` field: it stores the ordered `turn: tuple[TurnElement, ...]` (`TurnElement = ReasoningTrace | TextPart | ToolCall`, emission order) and derives the `.text`/`.tool_calls` views; the assistant turn's re-feedable text is its `TextPart` elements, read as `.text`. `UserMessage` and `AssistantMessage` take their single argument positionally (`UserMessage("hi")`, `AssistantMessage("hey")`), and a bare string `turn` coerces to one `TextPart` on every construction path. `output` is the generation result payload `OutputT` on `Response.output`/`ProviderResult.output` (and the adapter's `_parsed_output`), the assistant text or the SDK-parsed `response_format` instance. `content` and `output` coincide only for plain text: a structured `output` is a parsed `BaseModel`, not a `Part`, so it can never appear in a message body, while the same turn's re-feedable text lives separately on `Response.assistant_message`. `reasoning` is opaque provider round-trip data, distinct from both: it is never a `Part` and never in a user or tool body (OpenAI Responses models it as a separate output item and the OTel GenAI convention as a token counter, neither as content); it is carried as `ReasoningTrace` elements in `AssistantMessage.turn`, each `reasoning` the producing SDK item's `model_dump(mode="python", exclude_none=True)` dict, and it is a dict, not a JSON string (unlike `args_json`), because neither provider's reasoning wire form is a string. At the tool layer, `ToolOutput` names what a tool function returns, the union `MessageContent | ToolOutputExplicit[AppDataT]`; it is distinct from the generation-layer `output`/`OutputT` the same way `Response` and `ProviderResult` are split by layer.

## Design rules

- Generation only via binding: `LLM` has no generate methods; `LLM.bind(...) -> BoundLLM[OutputT]` freezes everything that determines the cacheable prompt prefix. No per-call parameter overrides; changing parameters is `rebind(...)`. `automatic_prompt_caching` is a required keyword of `bind` with no default: the library never chooses a caching configuration for the user, because caching changes billing, and any convenience on top of user-stated caching is opt-in and default-off. A `False` default was proposed and rejected: an unstated `False` silently opts every high-volume binding out of caching, a billing choice as real as opting in, so every call site states the boolean.
- Prompt caching is the bound `automatic_prompt_caching` plus user-placed `cache_breakpoint` marks, honored under either binding value; bound False with no marks caches nothing. `system_prompt` binds as a `str` or a `Sequence[TextPart]` (frozen to a tuple; empty raises `ValueError`): the parts form exists to carry marks inside the frozen prefix, rendered by the anthropic adapter as one system block per part and by the openai adapter as a developer-role input message first in every request's input (the SDK documents `instructions` as "a system (or developer) message inserted into the model's context", and only input message parts carry `prompt_cache_breakpoint`). The anthropic adapter writes at most 4 `cache_control` markers per request (the API limit): the binding's own markers (marked system parts, the automatic frozen-prefix and last-message markers) spend slots first and over-spending raises `ValueError` at bind, then the latest marked message parts fill the remainder, older marks left unwritten, mirroring openai's documented server rule (latest 3 explicit written in implicit mode, latest 4 in explicit mode, older breakpoints read-only), so a conversation that accrues one mark per turn keeps working on both providers. A marked part in a `ToolMessage` must be the message's last part: the marker goes on the enclosing `tool_result` block (the placement the API documents), which is equivalent only there; any other position raises `AbortBatchError` instead of silently moving the boundary. `AssistantMessage` rejects a marked turn `TextPart` at validation because openai's assistant replay text param carries no `prompt_cache_breakpoint` key, so the mark would be provider-divergent. The adapter never sends anthropic's top-level automatic `cache_control` request param: it is unavailable on Bedrock, which this adapter also serves. The marker TTL is the adapter's `cache_ttl: Literal["5m", "1h"] = "5m"` (exposed on `anthropic_model` and `anthropic_bedrock_model`), applied uniformly to every marker the adapter writes: "5m" omits the ttl key (the API default, keeping the wire form byte-identical to before `cache_ttl` existed) and "1h" writes `ttl: "1h"`, billing writes at 2x via `cache_write_1h_usd_per_million_tokens`. The openai adapter has no `cache_ttl` because "30m" is `prompt_cache_options.ttl`'s only value. `generate_many` takes `warm_cache: bool = False`, opt-in per the convenience rule above: it runs conversations[0] to completion before admitting the rest, because a cache entry is readable only after the response that writes it begins, so parallel identical prefixes each pay a full cache write. It warms unconditionally (never inspecting whether the binding caches), a first item ending in a `GenerationError` still admits the rest (a rejected 200 wrote the prefix; after a transport failure the rest run against a cold cache, with no second warmer), an `AbortBatchError` from the first item raises before any sibling starts, and there is no warmup ladder. A per-part TTL is rejected until someone asks: the API allows mixing TTLs in one request but requires the 1h markers before the 5m markers (a docs-only rule, not verifiable by SDK introspection: https://platform.claude.com/docs/en/build-with-claude/prompt-caching, "Mixing different TTLs"), an ordering-validation surface for a pattern the uniform `cache_ttl`'s two-binding workaround already covers.
- The application owns the tool loop; langchaint ships no library-owned agent loop. A tool function returns data (`str | Sequence[Part]`, or a `ToolOutputExplicit` wrapping that plus `is_error` and `app_data`) and nothing that steers control flow: stop, route, escalate, and needs-approval are decisions the app makes in its own loop between turns, reading `app_data` or `is_error`. Tool content is model-facing, so it is exactly `str | Sequence[Part]`, the same type as `ToolMessage.content`: it must already be in a form the model reads, so a function with a typed result serializes it to that form itself (a single `TextPart`/`ImagePart` is returned as a one-element `Sequence[Part]`, since a bare `Part` is not a `Sequence[Part]`). Content carries no `BaseModel`/`Mapping` arm and `dispatch` does no serialization: model-facing content is not the place for a typed object, unlike the `app_data` channel, which is app-facing (the model never sees it) and does carry a typed `BaseModel` or `Mapping` through untouched. A control-flow return channel is forbidden (as in LangGraph's `Command`, whose `goto` reroutes the loop and `update` mutates engine-owned state through a leaf tool's return value): it is the symptom of moving the loop out of the app, which langchaint does not do. A tool comes in two forms a `ToolManager` holds together through the `DispatchableTool` protocol (a `name`, a `schema()`, a `dispatch()`): `Tool`, whose `args_model` pydantic model is both schema source and argument validator, and `RawSchemaTool`, for a tool whose schema is a raw JSON schema and not a pydantic model (an MCP tool discovered at run time). `RawSchemaTool` carries `args_schema` (the JSON schema, sent to the provider unchanged, already the `model_json_schema`-shaped `Mapping` the adapters send) and a function taking the parsed arguments as a `dict[str, object]` (the type dispatch builds; a `Mapping[str, object]` annotation is also accepted by contravariance); it validates only that the arguments are a JSON object (via a shared `TypeAdapter(dict[str, object])`, reusing the `DispatchInvalidToolArgs` outcome so a non-object or malformed `args_json` stays model-correctable) and defers every field-level rule to the tool's owner, because the MCP server re-validates its own inputs and requiring a hand-written pydantic model per discovered tool would defeat run-time discovery. `RawSchemaTool` exists only because the wire path was already schema-based (adapters read `ToolSchema.args_schema`, never `args_model`); it adds no adapter code. `RawSchemaTool` has no `validate_and_run`: that seam exists on `Tool` because the validated arguments reach the caller typed, and a `RawSchemaTool` caller gains nothing over parsing the JSON itself. `ToolManager.dispatch_many(tool_calls)` is the concurrent counterpart of `dispatch` for the several tool calls of one assistant turn: every call starts at once and the outcomes return ordered by `tool_calls` position; it steers no control flow, so it sits inside the app-owned loop, not in place of it. Its failure rule extends dispatch's exceptions-are-defects rule to a batch: every call settles first, then the defects raise together as one `DispatchExceptionGroup` (an `ExceptionGroup` subclass) whose `completed_outcomes` carries the settled calls' outcomes (each naming its call via `tool_message.tool_call_id`), so `app_data` a completed sibling produced (a billing record for money the tool spent) survives the raise, the same principle as `GenerationError` preserving a rejected 200's billing on `attempt_records`; `derive` keeps `completed_outcomes` on the subgroups `except*` splits off. Cancellation is never grouped: a non-Exception `BaseException` a sibling produces re-raises bare (`ExceptionGroup` rejects such members, and grouping a `CancelledError` would swallow cancellation), and cancelling the enclosing task cancels the sibling dispatches and re-raises only after they finish unwinding (a bare `asyncio.gather` cancels its children without awaiting them, so `dispatch_many` settles them itself before propagating). A defect co-occurring with a sibling's own `CancelledError` still surfaces (review adoption): the bare re-raise chains a `DispatchExceptionGroup` as its `__cause__`, carrying the defects and `completed_outcomes` as usual, so cancellation winning the raise never silently discards a defect or a settled sibling's billing. There is no concurrency-limit parameter until someone asks.
- One `RateLimiter` (`rate_limiter.py`) owns retrying and pacing: `max_attempts`, `backoff_base_seconds`, `backoff_max_seconds`, `max_in_flight`. It is stateful and shareable: one instance passed to several `LLM`s is one shared budget for the account they hit. Its slot gates every request-start path and backoff sleeps outside the slot. There is deliberately no `requests_per_minute`: an in-flight bound self-adjusts throughput along request duration.
- Rate-limit errors gate admission account-wide. `Provider.classify` returns `"rate_limit"` for them (anthropic 429 and 529, openai 429), which lands on `TransientError.is_rate_limit`; a server-stated retry-after also counts as rate-limit evidence. The limiter then pauses admission for everyone sharing it: for the server-stated wait when one was sent (capped at 60 seconds so an erroneous or hostile header cannot stall the client indefinitely), else for the failing task's backoff delay. After the pause, admission stays limited to one probe request at a time until a completed request records success. Other transient errors (timeouts, 5xx) pause nobody; the failing task's own backoff covers them.
- Adapters wrap official SDK clients and delegate stream assembly and structured-output parsing to the SDK. No hand-written wire TypedDicts.
- A structured parse that returns no parsed output is classified three ways by the adapter's `_parsed_output`. A provider-signaled refusal (anthropic `stop_reason == "refusal"`, openai a `ResponseOutputRefusal` content block) raises `RefusalError`; a truncation at the token cap (anthropic `stop_reason == "max_tokens"`, openai `status == "incomplete"` with `incomplete_details.reason == "max_output_tokens"`) raises `ExceededMaxCompletionTokensError`; every other empty parse raises `TransientError`, whose exhaustion surfaces `RetriesExhaustedError`, never silently-wrong data. `RefusalError` and `ExceededMaxCompletionTokensError` are per-item terminal results, not batch-abort: each becomes a `to_row` failure row like `RetriesExhaustedError` and leaves siblings running. Neither is retried by default: an app whose economics differ on refusals overrides `_parsed_output` (the "subclass the adapter" escape hatch), and the token-cap fix is a larger `max_completion_tokens` via `rebind`. The text path never raises for these: a capped or refused text turn is a legitimate `Response` the app inspects by `stop_reason` (`"max_tokens"` / `"refusal"`); the two errors fire only on the structured path.
- The error taxonomy splits two orthogonal properties. The retry axis (read by the loop in `llm.py`): the loop retries a `TransientError` and propagates everything else, so "non-retriable" is a concept, not a class. The batch axis (read by `generate_many`): `AbortBatchError` (what `Provider.classify` returns `"abort"` for) cancels the sibling requests, while a `GenerationError` becomes one item's failure row. `AbortBatchError` and every `GenerationError` leaf are non-retriable and differ only on this batch axis. `GenerationError` is the base for the three terminal per-item results (`RetriesExhaustedError`, `RefusalError`, `ExceededMaxCompletionTokensError`); it carries `attempt_records`, `model`, `provider_name`, `elapsed_seconds`, and `stop_reason`, and derives `usage` (the paid total, carrying `cost_in_usd`) as the fieldwise sum of the records. The adapter that detects a refusal or truncation raises the leaf carrying only the one rejected 200's billing (its `usage` and `usage_raw`); the retry loop records that attempt and re-raises the enriched leaf, so no unenriched leaf reaches a caller. `AttemptRecord` carries `usage` (the attempt's billing with `cost_in_usd` inside, `ZERO_USAGE` for a transport failure that billed nothing) and `usage_raw` (the raw SDK usage, `None` when no wire payload existed, the "did this attempt reach a 200" signal), so a rejected 200's cost survives on the failure row. All exceptions live in the bottom-level `exceptions.py`, which imports `StopReason`, `Usage`, and `ZERO_USAGE` (leaf modules) plus pydantic `BaseModel` (the `usage_raw` type) at runtime and stays acyclic; `response.py` imports the error and record types from it. `DispatchExceptionGroup.completed_outcomes` is typed with the dispatch outcome types from `tools.py` (which imports `exceptions.py` at runtime), so that import is `TYPE_CHECKING`-only with quoted annotations, keeping the runtime import graph acyclic.
- `Usage` carries exactly two kinds of data, and this is the criterion for any future field: provider-invariant counters (the three-way input partition, `output_tokens`, `output_tokens_reasoning`; a new provider either fits this partition or does not belong in `Usage`), and `cost_in_usd`, the adapter-priced scalar that absorbs provider-variant billing structure (TTL rate splits, tiers) priced once in the adapter. Provider-specific detail in counts is never modeled in `Usage`; it stays on `usage_raw`. Cost is stored, never derived (the counters collapse the 5m/1h write split pricing needs), and no validator cross-checks it. `Usage` is summable: `__add__` folds every field including cost, `ZERO_USAGE` is the additive identity and the `sum(..., start=ZERO_USAGE)` seed, and `Usage.sum_of(usages)` is the import-free cross-`Response` aggregator. `usage` is never `None` on any carrier (`AttemptRecord`/`TransientError`/`ProviderResult`): a non-billing attempt is `ZERO_USAGE`, and `usage_raw is None` is the "no wire payload" signal instead.
- Usage scope: `usage` is the paid total across every attempt of the call, on both `Response` and `GenerationError`, so the two mean the same thing and the default a caller bills on is the money actually spent. `Response.usage_successful_attempt` (Response only, a `GenerationError` kept no success) is the single kept answer's own usage, the one matching `output` and `raw.usage`. Both are properties folded from `attempt_records`, the one source of truth (`usage` sums the records, `usage_successful_attempt` reads the last), so `Response` stores no usage field and the two cannot desync; they diverge only when a billed 200 was retried (an empty structured parse retried as transient). `to_row` has one paid-total cost column (`cost_in_usd` reads `usage.cost_in_usd`), and the usage-counter columns are likewise paid totals across attempts. The tracing mapper reads `result.usage.*`, so span usage and cost are all-attempts totals, with per-attempt detail in the `langchaint.attempt_failed` events.
- Bedrock support comes from the SDKs' bundled clients, not from a Converse adapter. Anthropic Bedrock is two disjoint SDK client classes, the legacy `InvokeModel` surface `AsyncAnthropicBedrock` and the Messages-API surface `AsyncAnthropicBedrockMantle`, and the catalog models split across them; the client class and the wire model id are per-model data in `ANTHROPIC_BEDROCK` (a `BedrockRouting` keyed by `AnthropicModelName`), because neither is derivable from the native name by a rule. OpenAI on Bedrock is `AsyncBedrockOpenAI` passed as `client`.
- OpenAI support is the Responses API only (`OpenAIResponsesProvider`); there is no Chat Completions adapter. Third-party chat-completions-compatible servers (vLLM, Ollama) are out of scope until someone asks.
- Streams yield `StreamItem = str | ToolCall`, nothing else. Each `ToolCall` is yielded once, complete, when its block closes; there are no tool-call delta items because partial argument JSON is not actionable. There are no usage or stop items: `usage` (carrying `cost_in_usd`) and `stop_reason` live on `final()`'s `Response`.
- No rewrap, no copy: never take data out of an SDK object only to reconstruct it in a package object of the same shape. Pass provider values through by reference; construct a package model only where the shape genuinely changes (normalizing `Usage`, deriving `StopReason`).
- Discriminate outcomes by type, not by a nullable flag: a function with several outcomes returns a union of frozen dataclasses, one arm per outcome, and the arm that carries extra data types it as a required non-optional field, so matching the arm narrows the payload and no consumer writes `cast`, `assert x is not None`, or `# type: ignore` to read a field the discriminator already guarantees is present. Split into arms only where the fields genuinely differ: two outcomes that carry the same fields and differ by a boolean stay one arm holding that boolean, not two field-identical arms. The rule governs a value a caller receives and branches on, not an element type gathered into a homogeneous collection and consumed by folding (filter, count, positional read). The rule also governs the payload type: `Tool` carries an `AppDataT` type parameter (defaulting to `None`) that the checker solves from the function's `ToolOutputExplicit` return, and `Tool.dispatch` returns `DispatchHandled[AppDataT]`, so an app that dispatched a known single tool reads `app_data` at its concrete type with no `isinstance`. The type is `AppDataT | None` because `app_data` is genuinely optional; a plain `is not None` guard is not the banned type-narrowing. `ToolManager.dispatch` holds a heterogeneous tool set whose per-call `AppDataT` is erased, so it surfaces `DispatchHandled[BaseModel | Mapping[str, object] | None]` and the `isinstance` on `app_data` there is the intended heterogeneous fold.
- Reasoning is preserved verbatim across turns. Each reasoning element the model produces (an anthropic `thinking`/`redacted_thinking` block, an openai `reasoning` item) becomes one `ReasoningTrace` in `AssistantMessage.turn`, in emission order; the consuming adapter re-emits every trace in place, unconditionally. A trace is replayable only through the provider that produced it: re-fed through the other provider, its dict's type key is outside that API's input union, and the resulting provider error is the intended loud failure of a wrong-provider replay (deliberate provider switching first rebuilds concluded assistant turns without their traces, the same mechanism as trimming). There is no `provider_name` tag on the trace and no adapter-side drop of another provider's traces: that drop was a silent trim, which the control-surface rule below reserves for the app, and it contradicted the surface-the-provider-error policy this rule ends with. Three motivations: quality (a reasoning model that cannot see its prior reasoning across the app-owned tool loop re-derives or contradicts itself), correctness (Anthropic 400s a tool-use continuation whose latest turn's thinking blocks are missing or modified; OpenAI's stateless replay under `store=False` requires output items re-sent in their original order, with the reasoning payload in `encrypted_content`, which the API populates only under `include=["reasoning.encrypted_content"]`, sent on every request), and prefix caching (reasoning sits inside the growing cached prefix, so hitting the cache across a tool loop requires it present and byte-identical every turn). The payload is one opaque dict per element, the SDK item's `model_dump(mode="python", exclude_none=True)`, re-fed on consume by a single contained `cast` to the SDK param and appended unchanged, never neutralized field-by-field (the two providers' reasoning shapes are disjoint and the string fields must round-trip byte-identical) and not a JSON string (neither provider's reasoning wire form is a string, unlike `args_json`). Full reasoning history is the default and the conversation is the only control surface: trimming is the app's job, by rebuilding concluded assistant turns without their traces; a turn whose tool calls still await results must keep its reasoning, and there is no bind-time on/off parameter because it would be redundant with editing the conversation. Replay is coupled to the binding: Anthropic rejects thinking re-fed on a request whose binding enables no reasoning, and the adapter surfaces that provider error rather than silencing it.
- Provider-side rules are the provider's to enforce. Never probe or test an endpoint to learn what error an invalid input combination raises, and never add client-side guards guessing at provider-side rules: send the request and surface the provider's error. Client-side raises are reserved for documented provider facts (the 4-marker limit, the tool_result marker placement) and for defects that would otherwise produce a silently wrong result.
- Pydantic only where serde plus validation pay for themselves. The qualifying models are: the message tree (`UserMessage`/`AssistantMessage`/`ToolMessage`/`TextPart`/`ImagePart`/`ToolCall`/`ReasoningTrace`), which validates the content and turn unions and coerces lists to tuples at the user-facing construction boundary and is what a caller persists to store a conversation (`AssistantMessage` is the ordered `turn` union; `TurnElement`'s members have disjoint field sets, so re-validation selects the member by field match without a discriminator); `Usage`, whose `NonNegativeInt` counters and partition cross-check validator guard the openai subtraction-derived partition; and tool args models, which validate model-produced argument JSON. A `RawSchemaTool` deliberately opts out of that field-level validation, checking only that the arguments are a JSON object and leaving the field rules to the tool's owner: validation there would require reconstructing a pydantic model from a JSON schema the library already has, and the MCP server re-validates anyway. Everything else is a frozen dataclass or NamedTuple.
- The SDKs are optional dependencies, one extra per backend (`langchaint[anthropic]`, `langchaint[openai]`, `langchaint[all]`); only `pydantic` is a hard dependency. The neutral core imports no SDK, so `import langchaint` needs neither package. Each backend lives in its own subpackage (`langchaint.anthropic`, `langchaint.openai`) that imports its SDK at module top, guarded so a missing package raises a `ModuleNotFoundError` naming the extra to install. The import path is the boundary: only code that reaches for a backend requires its SDK, and a type checker following `langchaint.openai` never resolves anthropic's types (so the catalog is split by backend, not shared). Each backend subpackage holds everything for that provider, the model catalog and the concrete adapter (`langchaint.anthropic.messages_provider`, `langchaint.openai.responses_provider`); the neutral base contract stays in `langchaint.provider` (singular) and imports no SDK.
- Each backend subpackage holds at least one function returning a ready `LLM` (`anthropic_model` in `langchaint.anthropic`, `openai_model` in `langchaint.openai`; anthropic adds the Bedrock sibling `anthropic_bedrock_model`), its `Literal` model-name type (`AnthropicModelName`/`OpenAIModelName`), and its pricing table (`ANTHROPIC_PRICING`/`OPENAI_PRICING`). Each function takes the provider's own model identifier as a `Literal` (the wire string is the model's only name), `client` defaulting to the native first-party SDK client built from environment credentials, `pricing` defaulting to the model's public prices looked up in the backend's table and overridable by passing a `PricingTable` of your own rates, and `rate_limiter`. The signatures are not symmetric: `anthropic_model` takes two keywords `openai_model` lacks, `default_max_completion_tokens: int = 4096` (the Messages API requires `max_tokens` and its adapter fills a `None` `InferenceParams.max_completion_tokens` with this default, while the Responses API needs no such default) and `cache_ttl: CacheTtl = "5m"` (the anthropic cache-marker TTL; openai's only TTL is "30m", so there is nothing to configure). The anthropic subpackage adds a second constructor, `anthropic_bedrock_model(model, *, aws_region=None, client=None, http_client=None, pricing=None, default_max_completion_tokens=4096, cache_ttl="5m", rate_limiter=None)`, for Bedrock, because Bedrock is a distinct surface with per-model wire model ids and two client classes, not a client swap on `anthropic_model`: it reads the model's `BedrockRouting` from `ANTHROPIC_BEDROCK`, builds the routing surface's client class (or, when a `client` is passed, validates its class matches the surface and raises `ValueError` on mismatch), and sends the routing's wire model id while defaulting `pricing` to `ANTHROPIC_PRICING[model]`, the same object `anthropic_model` uses, keyed by the native name. `http_client` passes a custom `httpx.AsyncClient` (loaded certs, a proxy) to the constructor's chosen client class, keeping the class routing a whole `client` would forgo; it is only for the default-client path, so passing both `client` and `http_client` raises `ValueError`. The native constructors need no such keyword: their single client class makes `client=AsyncAnthropic(http_client=...)` lossless, while the Bedrock constructor picks one of two classes from the model. The two Bedrock client classes override `copy()` without the base `AsyncAnthropic.copy`'s `http_client or self._client` reuse (anthropic 0.116.0), so the adapter's `with_options(max_retries=0)` copy re-feeds `client._client` to keep a custom transport that a plain copy would drop. Models outside the catalog are built directly from a concrete adapter re-exported by the backend subpackage: `from langchaint.anthropic import AnthropicMessagesProvider; LLM(AnthropicMessagesProvider(...))`. Prices are the one provider fact not verifiable by SDK introspection; each subpackage docstring carries the source URL.
- The public surface is tiered by audience: applications import the neutral core from top-level `langchaint` and each backend's constructor, names, pricing, and concrete adapter from its subpackage (`langchaint.anthropic`/`langchaint.openai`); adapter authors import the provider contract from `langchaint.provider` (singular: `Provider`/`BoundProvider`/`ProviderStream`/`ProviderResult`/`Binding`/`ErrorClass`). Top-level `__all__` re-exports only the SDK-free application surface, so the provider constructors, pricing tables, and concrete adapters are off it (re-exporting them would force `import langchaint` through both SDKs). Internal helpers (`Admission`, `SequenceNotStr`, `ToolSchema`, `DispatchableTool`) and the adapter-author contract stay importable from their own modules but off it (`RawSchemaTool` is an application-facing constructor and is on the top-level surface beside `Tool`).
- `InferenceParams` (`inference_params.py`) is deliberately minimal: only `max_completion_tokens`, `reasoning_effort`, and `temperature`, with no `top_p` or `seed`. `temperature` is `float | None = None`, None leaving the provider default in place (each adapter maps None to its SDK's omit sentinel; both `messages.create` and `responses.create` take `temperature` on the wire). An unmapped provider parameter is reached by subclassing the adapter, never a passthrough dict.
- `to_row(Response | GenerationError) -> dict[str, RowValue]` (`response.py`) flattens many results and errors into uniform rows for logging or dataframes; a success and any terminal failure (retries exhausted, refusal, truncation) fill the same keys.
- OTel tracing ships in-tree as `langchaint.tracing` under a `langchaint[otel]` extra (opentelemetry-api only; the SDK is a dev dependency for the tests' in-memory exporter), thin and content-free, following the guarded-import subpackage pattern and off the top-level `__all__`. The governing premise: the wrapper never fakes an event boundary a span is supposed to measure. So `TracedStreamHandle` iterates (recording `langchaint.time_to_first_token_seconds` and closing the span on a failing or abandoned stream) rather than pass through the iterator it needs to witness. `TracedBoundLLM.generate_many` is the deliberate exception: it delegates to `BoundLLM.generate_many` under one aggregate INTERNAL span carrying only `langchaint.batch_item_count`, and leaves per-item cost and timing to the returned rows (`to_row`); the mapper is not invoked for a batch. Span name (`{operation} {model}`, operation `"chat"`), kind, and status are wrapper-owned; only attribute names and values are mapper-owned, resolved once from `attribute_format: AttributeFormat | AttributeMapper` (default `"gen_ai"`). Kind is CLIENT for a span that wraps one outbound call (`generate_one`, the stream span) and INTERNAL for `generate_many`'s aggregate span, which makes no call of its own. The mapper receives `Response[object] | GenerationError` and never the conversation, so no built-in mapper can leak a prompt (content capture requires subclassing `TracedBoundLLM`); it is called only under `span.is_recording()`, and a raising mapper is caught and logged on the `langchaint.tracing` logger, never propagated, so a telemetry bug never discards a paid result. Wrapping is unconditional (`TracedLLM` mirrors `bind`/`rebind` so a rebound object stays traced); enabling, disabling, or routing tracing is OTel SDK configuration, never application code. Retry visibility is span events (`langchaint.attempt_failed` per failed record), not child spans. GenAI convention keys are verified against the pinned `opentelemetry-semantic-conventions` revision named in the module comment (0.64b0), never asserted from memory; the three-way cache partition stays whole under `langchaint.*` because the convention has no `cache_none` counterpart. `pyrefly` includes `tests` on its `search-path` so `tests.test_tracing` reuses `tests.test_bound_llm`'s fakes.
- Verified provider facts (checked against anthropic 0.116.0 / openai 2.45.0): anthropic `input_tokens` excludes cache tokens, openai Responses usage `input_tokens` includes `input_tokens_details.cached_tokens` and `.cache_write_tokens`; anthropic `Usage.output_tokens_details` is optional (`OutputTokensDetails | None`) with a required `thinking_tokens: int`, while openai `ResponseUsage.output_tokens_details` is required with a required `reasoning_tokens: int` (this asymmetry is why the anthropic adapter defaults `output_tokens_reasoning` to 0 when the details object is absent and the openai adapter reads the counter directly); anthropic splits cache writes into 5m/1h (`CacheCreation`); openai bills cache writes starting with GPT-5.6; on gpt-5.6+ openai caching is configurable per request (`prompt_cache_options` with `mode: "implicit" | "explicit"` and `ttl: "30m"`; explicit mode with no breakpoints disables caching) and input text/image parts take `prompt_cache_breakpoint`, up to four written per request; the Responses API stores responses server-side unless `store=False` is sent; both SDK clients retry internally by default (`DEFAULT_MAX_RETRIES = 2`) and honor `retry-after-ms`/`retry-after` only up to 60 seconds; both SDKs' `APIStatusError` exposes `.response` (an httpx.Response) whose headers carry retry-after; `with_options(max_retries=0)` returns a client copy on all five client classes (`AsyncAnthropic`, `AsyncOpenAI`, `AsyncBedrockOpenAI`, `AsyncAnthropicBedrock`, and `AsyncAnthropicBedrockMantle`). Anthropic Bedrock is two disjoint client classes (neither subclasses the other): `AsyncAnthropicBedrock`, the legacy `InvokeModel` surface, and `AsyncAnthropicBedrockMantle`, the Messages-API "Claude in Amazon Bedrock" surface; both expose `messages.create`/`.parse`/`.stream` and construct offline from `aws_region` alone. Reasoning shapes (introspection-verified): anthropic `ThinkingBlock` is `{signature, thinking}` and `RedactedThinkingBlock` `{data}`, with `ThinkingBlockParam`/`RedactedThinkingBlockParam` members of the `MessageParam.content` union; openai `ResponseReasoningItem` is `{id, summary, content, encrypted_content, status}`, a `response.output` union member, and `ResponseReasoningItemParam` a `ResponseInputItemParam` member; `"reasoning.encrypted_content"` is a `ResponseIncludable` member and `include` is a parameter of `responses.create`/`.parse`/`.stream`. Reasoning runtime behavior: anthropic requires the latest assistant turn's thinking blocks re-sent unmodified in a tool-use continuation (else 400), filters prior turns' blocks itself, cannot have consecutive thinking blocks rearranged, and rejects thinking re-fed on a request whose binding enables no reasoning; openai requires output items between the last user message and the function-call outputs replayed in original order, and populates `encrypted_content` only when the request carries `include=["reasoning.encrypted_content"]` (`store=False` alone returns it None). OTel GenAI semconv 0.64b0 defines no reasoning message part, only the counter `gen_ai.usage.reasoning.output_tokens`.

## Module map

One line per module; the list changes rarely, so it does not rot the way a signature snapshot would.

- `llm.py`: `LLM` composes a provider adapter and a `RateLimiter`; `LLM.bind(...)` freezes the cacheable prompt prefix into a `BoundLLM`.
- `provider.py`: dual-audience. It holds the adapter-author contract (`Provider`/`BoundProvider`/`ProviderStream`/`ProviderResult`/`Binding`/`ErrorClass`) alongside four types that stay on the application surface (`StreamItem`/`ToolChoice`/`SpecificTool`/`PricingTable`). It is the neutral base contract and imports no SDK; the concrete adapters live in the backend subpackages.
- `rate_limiter.py`: `RateLimiter`, the single owner of retrying and pacing.
- `exceptions.py`: the whole error vocabulary: `TransientError`, `AbortBatchError`, `AttemptRecord`, the `GenerationError` base and its leaves (`RetriesExhaustedError`, `RefusalError`, `ExceededMaxCompletionTokensError`), `InvalidToolArgsError`, `DispatchExceptionGroup`, and `StreamProtocolError`.
- `response.py`: the generate result `Response`, plus `to_row` for flattening a `Response` or a `GenerationError` to one row shape.
- `streaming.py`: `StreamHandle`, an async iterator of stream items whose `final()` returns the `Response`.
- `tools.py`: `Tool` (pydantic `args_model`) and `RawSchemaTool` (raw JSON schema, arguments validated only as a JSON object) unified by the `DispatchableTool` protocol, `ToolManager` (per-call `dispatch` and the concurrent `dispatch_many`), the tool return union `ToolOutput` and its explicit arm `ToolOutputExplicit`, and the dispatch outcome types (`DispatchHandled`/`DispatchInvalidToolArgs`/`DispatchUnknownTool`, union `DispatchOutcome`).
- `messages.py`: the provider-neutral message tree and content parts; `AssistantMessage` stores the ordered `turn` of `ReasoningTrace`/`TextPart`/`ToolCall` elements.
- `usage.py`: `Usage`, the token-count partition plus the adapter-priced `cost_in_usd`; `ZERO_USAGE`, its additive identity.
- `anthropic/`: the anthropic backend subpackage, everything that needs the anthropic SDK in one place. `__init__.py` holds `anthropic_model`, `anthropic_bedrock_model`, `ANTHROPIC_PRICING`, `ANTHROPIC_BEDROCK`, `BedrockRouting`, `AnthropicModelName` and re-exports the adapter and `CacheTtl`; `messages_provider.py` holds the concrete adapter `AnthropicMessagesProvider`. Importing the subpackage requires the anthropic SDK.
- `openai/`: the openai backend subpackage, everything that needs the openai SDK in one place. `__init__.py` holds `openai_model`, `OPENAI_PRICING`, `OpenAIModelName` and re-exports the adapter; `responses_provider.py` holds the concrete adapter `OpenAIResponsesProvider`. Importing the subpackage requires the openai SDK.
- `inference_params.py`: `InferenceParams` and `ReasoningEffort`.
- `tracing/`: the OTel tracing subpackage, everything that needs opentelemetry-api in one place. `__init__.py` holds `TracedLLM`/`TracedBoundLLM`/`TracedStreamHandle`, the `AttributeFormat`/`AttributeMapper`/`SpanAttributes` types, and the built-in `"gen_ai"` mapper. Importing the subpackage requires opentelemetry-api (install `langchaint[otel]`); it is off the top-level `__all__` so `import langchaint` stays OTel-free.

# Commit Review

After each commit lands, spawn one clean-context review agent for that commit. The agent has no memory of the writing session.
Use Sonnet as a reviewer for simple work, Opus for work that is more open to design objections (see point 8 below).

1. Scope: exactly one commit. Read ONLY from the git object store: `git show <sha>`, `git diff <sha>^ <sha>`, and `git show <sha>^:<path>` for surrounding context (for the root commit, `git show --root <sha>`). Never read the working tree or `HEAD`, which may have moved since the commit. Begin the review by printing the resolved hash, subject, and parent hash as proof of scope. Every finding must be about this commit's diff, not other commits, not the working tree, not the broader plan; design objections (rule 8) are exempt and may challenge a premise the commit inherits.
2. Do not re-run checks. `scripts/CI.sh` (`pyrefly check`, `ruff check`, and `pytest`) passed at zero errors on a green tree immediately before the commit was created, so re-execution is redundant, including the tests. Use static reading plus cheap read-only greps only.
3. Spec fidelity: before reading the diff, read the rules in `CLAUDE.md`, and state what the code should do. Then check the commit against your own reading, not the author's stated intent.
4. Depth: review correctness of code, adherence to the naming rules (`exact_name=exact_name`, explicit names), faithfulness of any stated provider facts, and consistency between the diff and the documents it edits (`CLAUDE.md`, `README.md`). Provider-behavior claims must be verified per the "Provider facts" section above, never asserted from memory.
5. Test quality: judge whether the added tests would catch a plausible wrong implementation, and whether they could be stronger. For adapter code, prefer tests that feed a constructed SDK object and inspect the normalized result over tests that mock the package's own types.
6. User experience: look for regressions in what application code has to write, especially added verbosity at public methods and constructors. Judge each public call site the commit changes by writing out the before and after forms; flag any that got longer or noisier without a stated reason.
7. Output: a report of confirmed issues, each with the evidence that confirms it, not a list of suspicions. Ground each finding in a concrete input and the wrong output it produces, or the exact `CLAUDE.md` sentence it violates. An empty report is a valid result.
8. Design objections (optional, non-blocking): after the confirmed issues, the reviewer may challenge the commit's design premises (SDK-delegation boundary, bind-only generation, the neutral ToolChoice vocabulary, retry ownership, usage normalization). Each objection must name (a) the specific premise it challenges, (b) the concrete cost if that premise is wrong, and (c) an alternative. Omit objections missing any of the three; do not restate preferences or re-litigate a premise the Design rules already record with their reasoning unless naming new evidence.
9. Done gate: a commit counts as Done only after its review has returned and any confirmed issues are fixed. One feature stays one commit: during the review -> correct loop, fixes amend into the reviewed commit (`git commit --amend`, force-pushing if it was already pushed), and the amended commit is re-reviewed. Design objections do not block the gate; when the review returns, the author answers each objection inline where the premise lives, in the relevant Design rule in `CLAUDE.md`: adopting it edits that rule, rejecting it appends the one-line reason for keeping the rule as written, which becomes rule 8's re-litigation shield. That edit rides in the same amend; there is no separate open-items file to keep in sync. Do not start the next unit of plan work on top of a commit that is not Done.

# Code smells

Whenever you find yourself writing the following code smells, ask "how can I redesign and refactor to avoid them?"

- `**kwarg` unpacking
- `Any` for typing
- `cast` as a general solution. Treat `cast` like Rust's `unsafe`: not banned, but never a reflex reach. The default is a redesign that makes the type flow without it. A `cast` is warranted only at a genuine boundary where a deliberately-opaque value re-enters a typed API and its shape is known-true by construction (it came from that same API's own serialization), the alternative is worse (for example a revalidation that silently reshapes the payload), and it is contained to that one line with a comment stating why the shape holds. Outside that narrow case, a `cast` is the smell to redesign away.
- Unpacking data from one container (list, dict, pydantic object) and back into another.
